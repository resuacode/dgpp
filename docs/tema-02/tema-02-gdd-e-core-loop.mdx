# Tema 2.1. Estructura del GDD y core gameplay loop

Este documento tiene dos objetivos prácticos: **saber qué secciones debe tener un GDD** y
**diseñar el bucle jugable principal**, que es el latido del juego. Ambos son el primer
trabajo real de diseño del proyecto.

## Objetivos

- Estructurar el GDD por bloques funcionales, alineado con el GDD transversal del módulo.
- Diseñar el *core gameplay loop* y sus loops secundarios.
- Distinguir loop de acción (segundos), loop de sesión (minutos) y loop de progresión (horas).
- Documentar el bucle de forma que sea prototipable.

## Contenidos

### 1. Anatomía del GDD

El GDD (*Game Design Document*) es el documento que traduce la visión en especificaciones. No
es un contrato inamovible: es un **documento vivo** que se actualiza con cada iteración. Un
GDD útil suele incluir, como mínimo:

- **Ficha técnica y *executive summary***: qué es el juego, género, plataforma, público.
- **Concepto y *x-features***: *high concept* y elementos diferenciadores.
- **Mecánicas y core loop**: reglas, controles y bucle principal.
- **Progresión y economía**: cómo avanza el jugador y qué recursos gestiona.
- **Contenido**: niveles, enemigos, objetos.
- **Arte y audio**: dirección estética y referencias.
- **Interfaz y UX**: HUD, menús, *feedback*.
- **Alcance y plan**: qué entra en cada iteración.

:::note[Alinéate con el GDD transversal]
La estructura obligatoria del módulo está en el [GDD transversal](../guia/gdd-transversal.mdx).
Este apartado desarrolla las secciones de mecánicas y core loop; el resto se completa en temas
posteriores.
:::

:::info[Ejemplo: documentación de *Spelunky*]
Derek Yu mantuvo notas de diseño y versiones del prototipo mientras el juego evolucionaba: el GDD
no era un PDF cerrado del día uno, sino un registro de decisiones. Ese enfoque encaja con la
estructura del [GDD transversal](../guia/gdd-transversal.mdx).
:::

### 2. Principios de un buen GDD

- **Vivo**: se versiona (v0.1, v0.2…) y refleja las decisiones tomadas.
- **Conciso**: mejor claro y breve que exhaustivo e inútil. El *one-page design* es un buen
  punto de partida.
- **Visual**: diagramas del loop, *mockups* de UI y tablas ahorran páginas de texto.
- **Accionable**: cada sección debe permitir que alguien construya algo.
- **De coordinación**: el GDD no es literatura de diseño; es la herramienta que alinea a
  programación, arte, audio y QA sobre la misma visión. Cada departamento debe encontrar en él
  lo que necesita para trabajar.

:::info[Ejemplo: *Spelunky*]
Derek Yu documentó iteraciones constantes de *Spelunky*: el GDD y los prototipos evolucionaron
juntos. Un documento vivo registra por qué se cambió la generación procedural o la economía de
tienda; un muro de texto inicial que nadie actualiza no guía al equipo cuando el juego real ya
es otro.
:::

### 3. Pensar las mecánicas por dimensiones

Listar mecánicas ("disparar", "saltar") sin organizarlas dificulta el balanceo y el *playtesting*.
Una forma útil de clasificarlas —complementaria al MDA— es por **dimensión del sistema lúdico**:

| Dimensión | Qué cubre | Ejemplos |
| --- | --- | --- |
| Control / interfaz | Entrada del jugador, respuesta del personaje | Apuntado, salto, agarre (VR) |
| Información | Qué se muestra, oculta o descubre | Niebla de guerra, pistas, HUD |
| Espacio | Geometría, rutas, zonas | Plataformas, cobertura, atajos |
| Recursos | Economía interna, inventario | Vida, munición, moneda |
| Tiempo | Ritmo, presión, turnos | Temporizador, *cooldowns*, día/noche |
| Social | Cooperación, competición, comunicación | Chat, emotes, PvP, trading |

:::info[Ejemplo: *Dead Cells*]
En *Dead Cells*, el **control** (combate rápido), el **espacio** (corredores y plataformas), los
**recursos** (celdas y armas entre runs) y el **tiempo** (presión de enemigos) se entrelazan: si
fallas en una dimensión, el loop se resiente. Clasificar tus mecánicas así revela huecos antes
de prototipar.
:::

:::tip[Revisión del GDD]
Recorre las mecánicas de tu proyecto y márcalas en esta tabla. Si una dimensión queda vacía,
pregunta si es intencional o si falta profundidad de diseño.
:::

### 4. El core gameplay loop

El **core loop** es la secuencia de acciones que el jugador repite constantemente y que
constituye la esencia del juego. Ejemplo genérico:

```text
Observar situación → Decidir acción → Ejecutar → Recibir feedback → Recompensa/consecuencia → (repetir)
```

Un core loop bien diseñado es **claro, satisfactorio y sostenible** en el tiempo. Conviene
distinguir tres escalas de bucle que conviven:

| Escala | Duración típica | Ejemplo |
| --- | --- | --- |
| Loop de acción | Segundos | Saltar, disparar, esquivar |
| Loop de sesión | Minutos | Completar un nivel, una oleada |
| Loop de progresión | Horas/días | Desbloquear mejoras, avanzar la historia |

:::info[Ejemplo: *Slay the Spire*]
El **loop de acción** dura segundos (jugar una carta); el de **sesión**, minutos (completar un
piso o morir); el de **progresión**, horas (desbloquear cartas entre runs). Las tres escalas
refuerzan el mismo núcleo —construir un mazo cada run— sin competir entre sí.
:::

:::tip[El *feedback* no es opcional]
Cada acción del loop necesita **retroalimentación perceptible** (visual, sonora, háptica). Sin
*feedback* claro, el jugador no aprende el sistema y el loop no engancha.
:::

### 5. Loops secundarios

Alrededor del core loop se construyen loops secundarios (colección, mejora, exploración,
social) que dan profundidad y motivan la rejugabilidad. Deben **reforzar** el core loop, no
competir con él por la atención del jugador.

:::info[Ejemplo: *Hades*]
El core loop es el combate por salas; los loops secundarios (conversaciones con personajes,
mejoras permanentes, relaciones) recompensan cada intento fallido sin sustituir el combate.
Si el diálogo obligara a pausar demasiado el loop de acción, la experiencia se fragmentaría.
:::

## Actividades

1. **Mapa del loop jugable**
   - Diagrama del core loop con sus acciones, *feedback* y recompensas, e identificación de al
     menos un loop secundario.
   - *Evidencia*: diagrama (imagen o esquema en texto) + breve explicación de cada paso.
2. **Checklist de secciones del GDD**
   - Verifica que el GDD v0.1 cubre las secciones mínimas y márcalas como "hecho / pendiente".
   - *Evidencia*: checklist completado en el GDD.
3. **Tabla de mecánicas por dimensiones** (refuerzo)
   - Clasifica las mecánicas primarias del proyecto según control, información, espacio, recursos,
     tiempo y social.
   - *Evidencia*: tabla en el GDD con al menos una mecánica por dimensión relevante o justificación
     de omisiones.
4. **Prueba del loop en papel** (refuerzo)
   - Simula el core loop con papel o piezas para validar que funciona antes de programarlo.
   - *Evidencia*: fotos o notas de la prueba y conclusiones.

## Evaluación

| Criterio | Insuficiente | Adecuado | Excelente |
| --- | --- | --- | --- |
| Claridad del core loop | Confuso o incompleto | Loop identificable | Loop claro, con feedback y recompensa |
| Estructura del GDD | Faltan secciones clave | Secciones mínimas presentes | Documento vivo, visual y accionable |
| Loops secundarios | Ausentes o contradictorios | Presentes | Refuerzan el core loop con intención |
| Coherencia doc–loop | Incoherente | Coherente | El documento permitiría prototipar |
